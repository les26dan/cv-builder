1. Tổng quan kiến trúc

Hệ thống hiện tại hỗ trợ người tìm việc tối ưu CV. Phần mới này bổ sung chiều ngược lại: Nhà tuyển dụng có thể upload kho CV ứng viên, sau đó nhập mô tả công việc để hệ thống tự động tìm và xếp hạng ứng viên phù hợp.

Corpus: Kho CV ứng viên do nhà tuyển dụng upload trước. Mỗi CV được tính embedding 1 lần duy nhất khi upload và cache lại → bước retrieval không phát sinh chi phí API bổ sung.

Triết lý thiết kế hai bước

Bước 1 — Chọn nhầm còn hơn bỏ sót

Lọc rộng, ưu tiên Recall. Chấp nhận false positive để đảm bảo không bỏ qua ứng viên phù hợp. Mục tiêu: Recall@20 ≥ 95%. Phương pháp nhanh, chi phí $0.

Bước 2 — Lọc chặt, xếp hạng chính xác

Từ 20 ứng viên, lọc ra top-5 tốt nhất. Ưu tiên Precision. Kết hợp đánh giá cấu trúc (không tốn token) + LLM batch reasoning. Mục tiêu: Precision@5 ≥ 80%.

2. Bước 1 — Hybrid Retrieval

1
Hybrid Retrieval: TF-IDF + Embedding
Input: corpus N CV + JD text → Output: top-20 CV

Công thức

Hybrid_score(cv, jd) = α × TF-IDF_score + (1 − α) × Embedding_score -- α = 0.4 (embedding được trọng số cao hơn: 60%) -- TF-IDF_score ∈ [0, 100] — cosine sparse vector -- Embedding_score ∈ [0, 100] — cosine dense vector 1536-dim
Metric chính: Recall@20 Ngưỡng mục tiêu ≥ 95% Latency: ~2ms Cost: $0.02/1k (chỉ tính embedding JD)

So sánh ba phương pháp retrieval

Phương pháp Recall@20 Latency p50 Cost/1k Ưu điểm Nhược điểm
TF-IDF (baseline) 0.87 2ms $0.00 Nhanh, free, chính xác từ khóa kỹ thuật Bỏ sót ngữ nghĩa tương đương
Embedding 0.92 1ms* $0.02 Hiểu ngữ nghĩa, đa ngôn ngữ Tốn chi phí embedding lần đầu
Hybrid (đề xuất) 0.96 2ms $0.02 Kết hợp tốt nhất cả hai Cần tune α

Cơ chế hoạt động từng phương pháp

🔤

TF-IDF (Lexical)

Token hóa văn bản → tính TF-IDF weight → cosine similarity trên sparse vector. Bắt chính xác từ khóa kỹ thuật: "Python", "Node.js", "AWS".

File: tfidfMatcher.ts

🧠

Embedding (Semantic)

Encode text → vector 1536-dim (text-embedding-3-small) → cosine similarity. Bắt được: "lập trình Python" ↔ "phát triển phần mềm".

File: embeddingMatcher.ts

⚡

Hybrid (Đề xuất)

Tổng hợp cả hai tín hiệu. α=0.4 nghĩa là embedding chiếm 60%, TF-IDF 40%. Lấy top-20 theo Hybrid_score.

New: ragSearchService.ts

3. Bước 2 — Structural + LLM Ranking

2
Ranking: Structural Score + LLM Batch
Input: top-20 từ bước 1 → Output: top-5 xếp hạng + lý do

2a. Structural Score (không tốn token)

Structural_score = w₁ × exp_years_match -- w₁ = 0.35 + w₂ × edu_level_match -- w₂ = 0.25 + w₃ × skill_coverage_ratio -- w₃ = 0.40 -- exp_years_match = 1 nếu yearsExp(CV) ≥ yearsRequired(JD), else graded -- edu_level_match = 0..1 theo thứ bậc: TC/CĐ < ĐH < Thạc sĩ < Tiến sĩ -- skill_coverage = |skills_CV ∩ skills_JD| / |skills_JD|
📅

Năm kinh nghiệm (w=0.35)

Tính từ experience[].startDate → endDate trong JSONB. So sánh với yêu cầu JD trích xuất bằng regex.

🎓

Bằng cấp (w=0.25)

Parse education[].degree, phân cấp 0–3. Graded nếu CV vượt yêu cầu.

🛠️

Tỉ lệ kỹ năng (w=0.40)

So sánh tập skills[] của CV với keyword list trích từ JD. Intersection / JD_skills_count.

2b. LLM Batch Ranking (1 lần gọi duy nhất)

Thay vì gọi LLM 20 lần riêng lẻ, hệ thống gom tất cả vào 1 prompt batch. Mỗi CV được rút gọn xuống ~200 token trước khi ghép.

// Prompt template (gpt-4o-mini, temperature=0, max_tokens=800) System: Bạn là chuyên gia tuyển dụng. Xếp hạng ứng viên theo JD. Trả về JSON array, không giải thích thêm. User: JD: {jd_text} Ứng viên (rút gọn ~200 tokens mỗi người): [1] CV-042: Senior Backend · Skills: Node.js, PostgreSQL, Docker · 5 năm KN [2] CV-017: Fullstack Dev · Skills: React, Python, AWS · 3 năm KN ... (20 ứng viên) Output JSON: [{"rank":1,"candidateId":"CV-042","score":87,"reason":"5 năm backend, thành thạo PostgreSQL"}]

Final Score

Final_score(cv, jd) = 0.3 × Structural_score + 0.7 × LLM_score -- LLM trọng số cao hơn vì hiểu ngữ cảnh sâu hơn -- Structural_score là "anchor" tránh LLM bị nhiễu bởi văn hoa mỹ
Metric chính: Precision@5 Mục tiêu ≥ 80% Precision@1 ≥ 85% Latency: ~1.2s (1 LLM call)

4. Thiết kế đánh giá (Evaluation)

📦

Dataset

400 cặp CV-JD có nhãn ground truth
200 positive (phù hợp) + 200 negative (không phù hợp)
25 ngành nghề · File: data/eval/pairs.json

🎭

Simulation

Với mỗi JD dương, corpus giả lập = 200 CV âm + 1 CV dương tương ứng (201 CV). Hệ thống phải tìm đúng CV dương trong 201 ứng viên. Lặp 200 lần.

🏷️

Ground Truth

Nhãn đã có sẵn — không cần gán nhãn thêm. Nhãn 1 = phù hợp, 0 = không phù hợp. LLM-assisted + manual review.

Ba câu hỏi nghiên cứu

RQ-R1
Hybrid Retrieval có đạt Recall@20 đủ cao để không bỏ sót ứng viên?
Metric: Recall@20
Mục tiêu: ≥ 95%
RQ-R2
Bước Ranking có cải thiện Precision@5 so với chỉ dùng Retrieval?
Metric: Precision@5, Precision@1
Baseline: top-20 trực tiếp
RQ-R3
Batch LLM tiết kiệm bao nhiêu token so với N lần gọi độc lập?
Metric: Token count, Cost/1k
Mục tiêu tiết kiệm: ≥ 60%
